# Hardware Compatibility

This table is generated by hand from `pyidotmatrix/capabilities.py`, the
single source of truth for what has actually been established about this
protocol and where. **It is never populated by runtime feature-probing** —
the device acks commands it doesn't act on (see
[Protocol Notes](protocol-notes.md#acks-confirm-receipt-not-effect)), so the
only trustworthy source is a dated hardware probe or a decompiled-source
citation, never "the SDK tried it and it seemed to work."

Primary hardware reference for every `32×32` row below: one panel, name
prefix `IDM-`, tested against APK `v2.1.2 googleRelease (2026-06-29)`.

Legend (matches `CapabilityStatus` in `pyidotmatrix/capabilities.py`):

| Symbol | `CapabilityStatus` | Meaning |
|---|---|---|
| ✅ | `VERIFIED` | Observed doing the right thing on real hardware. |
| ⚠ | `SOURCE_DERIVED` | Byte layout confirmed from the decompiled vendor app; never (or not conclusively) exercised on hardware. |
| ❓ | `UNKNOWN` | Wire bytes exist but their meaning or effect is unestablished even in the app source. |
| ✖ | `KNOWN_BROKEN` | Sent to real hardware and observed **not** working there. |

Look any of these up in code:

```python
from pyidotmatrix import capability
capability("text.show").status      # CapabilityStatus.VERIFIED
capability("text.show").evidence    # the probe/date/source behind that status
```

## How to extend this table

You don't need to write any SDK code to move a row from ⚠/❓ to ✅ — a
hardware probe on a panel we don't have is a first-class contribution (see
[CONTRIBUTING.md](../CONTRIBUTING.md)). The process:

1. **Confirm the wire format** first, from `pyidotmatrix/protocol/*.py` or
   the [reverse-engineering notes](reverse-engineering/) — don't guess bytes.
2. **Run it against your panel** and record what actually happened, good or
   bad. A probe that proves a feature *doesn't* work is exactly as valuable
   as one that proves it does — several rows in this table exist because a
   probe disproved an earlier assumption (`effect.speed`,
   `common.freeze_screen`, `common.set_speed`).
3. **Add a byte-exact regression test** if the finding changes a builder's
   behavior.
4. **Update the entry in `pyidotmatrix/capabilities.py`**, citing your probe
   script and the date. That's what this page renders from — there's nothing
   else to edit to get your result into this table.

See [CONTRIBUTING.md](../CONTRIBUTING.md) for how to run probes safely (some
are destructive, none belong in CI), and
[docs/PROBE_PLAN.md](PROBE_PLAN.md) for the open questions currently worth
panel time.
